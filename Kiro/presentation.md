## 第2章：背景とKiroの概念 (Background & Concept)

### 2.0. セクションヘッダー

**第2章：背景とKiroの概念**

---

### 2-1. 発表者の開発体験の変遷（Copilot → Q CLI → Kiro）

**初期：ツールを触り始める（2025年3月頃）**
* GitHub Copilotで生成AI開発に触れる
* コード補完やチャット形式が主流でまだ補助レベル

**中期：生成AIでコードを修正する（2025年5月頃）**
* Q Developer CLIを使いはじめる
  * プロジェクトソースを読み込んで理解した上でソースを作成できた
* MCP連携をはじめる
  * Backlog MCPを使ってチケット駆動開発を実施

**課題:**
* **"Vibe Coding"（雰囲気開発）** は衝撃的だったが、**長期的な開発の一貫性** に課題があった
* AIは**昨日の作業を覚えていない！**
* 既存のコードと整合性のないコードを生成する

**後期：Kiroを使う（2025年8月頃）**
* **仕様駆動開発**というコンセプトがこれまでの開発の弱点を解決する手法になるのではないかと感じる
* コミュニティも盛り上がっており、Waitlist待ちが発生するほどの注目度
* 使ってみて、課題感はあるものの、おおむね満足の開発体験を得ることができた

---

### 2-2. Kiroとは？：「チャット」ではなく「仕様」で対話する

Kiroは、「IDEにAIが入った」というより、「AIを中心にIDEを再設計した」ツールです。

**他のツールとの違い:**
* GitHub Copilot / Q Developer: コード補完・チャット型プラグイン（その場で完結）
* Cursor / Windsurf: 自律的なコーディングエージェント型IDE
* **Kiro: 「仕様書（Markdown）を書くこと」を中心にする**

**3つのドキュメントと共通コンテキストで開発が進む:**

1. **Requirements (要件):** ユーザーストーリーと受け入れ基準
2. **Design (設計):** アーキテクチャ、API、DB設計
3. **Tasks (実装計画):** 具体的な実装ステップ
4. **Steering（共通コンテキスト）** プロジェクトの説明や技術情報

**ポイント:**
* 人間がこのドキュメントを承認（Accept）するまで、AIはコードを書き始めない
* 実装計画をもとに作業を進める
* ソースコードと同列に「ドキュメント」を育てていく体験が得られる
* ドキュメントがプロジェクトの「記憶」として機能

---

### 2-3. 従来の開発との違い：「Vibe Coding」から「Spec-driven」へ

**従来のAI開発（Vibe Coding）:**
* チャットで「こんな感じで作って」と指示
* その場では動きますが、翌日には前日の文脈を忘れている
* 長期開発では一貫性が保てず、「作っては壊し」の繰り返しになる

**Kiroの開発（Spec-driven）:**
* まず要件・設計を文書化し、人間が承認してから実装を開始する
* ドキュメントがプロジェクトの「記憶」として機能する
* 新しい機能を追加する際も、既存の仕様を参照して一貫性を保つ

**非エンジニア向けの例え:**

* 従来：「昨日の議事録まとめておいて」 → 毎回フォーマットやまとめ方の癖が違う
* Kiro："議事録テンプレート" と "過去の会議方針" を共有 → AIが自動的に整合性を取る

---

### 2-4. アーキテクチャ

「Hello World」レベルでは検証にならないため、本格的なアーキテクチャを採用しました。

* **技術スタック:**
  * **アプリ:** Expo (React Native) で iOS/Android 両対応
  * **認証:** Amazon Cognito
  * **API:** API Gateway + Lambda (FastAPI on LWA)
  * **DB:** DynamoDB (Single Table Design)
  * **インフラ:** AWS（CDK）

* **開発プロセスのこだわり:**
  * **モノレポ構成:** フロント・バック・インフラを単一リポジトリで管理
  * **ADR (Architecture Decision Record):** 「なぜその設計か」を記録し、Kiroに読ませる
  * **OpenAPI:** API仕様書を事前に定義し、実装の契約とする

---

### 2-5. 付加作業

**通常の開発では後回しにされがちな作業**も開発中に改善作業として組み込みました。

* **テストコード実装:** 各サブプロジェクト単位でテストコードを実装
* **CI/CD:** テスト実行/開発環境へのデプロイ
* **Makefileによる共通スクリプト化:** テスト実行やローカルサーバ起動をワンコマンドで実行
* **テストデータのseed管理:** ローカル環境のデータ管理を一元化

### 3-1. Kiroの画面構成とドキュメント（ライブデモ 5分）

**実際のKiro IDEを見てみましょう。**

---

#### 画面の構成

* **左側:** プロジェクトのファイルツリー
  * 通常のソースコード（`/frontend`, `/backend`, `/infra`）
  * **`.kiro/`フォルダ:**
    * `specs`, `steering`

* **中央:** エディタ（コードとドキュメントを編集）
* **右側:** AIとのチャット画面
* **下部:** Terminal

---

#### 仕様ファイル（specs）

Kiroが作業の中心に置く3つのMarkdownファイルです。

1. **`requirements.md`**
   * ユーザーストーリー、受け入れ基準
   * 例：「ユーザーは対局結果を登録できる」「順位と点数を入力すると自動計算される」

2. **`design.md`**
   * システムアーキテクチャ、API設計、DB設計
   * 例：「DynamoDB Single Table Design」「Cognito認証フロー」

3. **`tasks.md`**
   * 実装タスクのリスト（TODO、In Progress、Done）
   * AIはこのタスクを見て、何を実装すべきか判断します

**ポイント:**
* これらのドキュメントは**AIが自動生成**するが、**人間が承認（Accept）**する必要がある
* ソースコードと同じように、ドキュメントもGitで管理される
* 作業途中に計画を修正したり、要件・設計の再定義を行うことも可能

---

#### コンテキストドキュメント（steering）

steeringディレクトリ配下のMarkdownフィアルは自動的に読み込まれます。
※下記は私の例。

- プロジェクトの内容に関するドキュメント
  - context.md - プロジェクトコンテキスト（概要、開発方針、重要な制約）
  - product.md - Janlogアプリ概要（主要機能、対象ユーザー、MVPスコープ）
  - structure.md - プロジェクト構造（モノレポ構成、ディレクトリ規約）
  - tech.md - 技術スタック（フロント/バック/インフラの技術選定とコマンド）
  - api-spec.md - API仕様書参照（OpenAPI仕様へのリンク）
- 開発ルールに関するドキュメント
  - scripts.md - スクリプト・コマンド体系（Makefile、開発コマンド、データベース管理）
  - environment.md - 環境分離戦略（local/development/production環境の設定）
  - spec-management.md - Spec管理運用ルール（features/improvements/bugfix/archive構造）

## 第4章：終わりに (Conclusion)

### 4-1. Kiroの課題とリアリティ：まだ完璧ではない

**成功例だけでなく、現実も共有します。**

1. **コンテキストロット**
   * 1セッション内での作業が長引くと、コンテキストが肥大化し、Rot（欠損）が発生します
   * 自身で開発した作業を忘れて整合性がおかしくなります
   * **対策:** 作業単位を小さくする

2. **未完了の状態で完了したと誤認する**
   * 未完成の状態で勝手に完了したと判断したりします
   * テストが失敗している状態でテストせず完了と判断します
   * 失敗が続くと難易度の高い作業をスキップしたり無視したりします
   * **対策:** AI任せにせず、人間が判断し、追加指示を行う

3. **既存コードの破壊**
   * 新機能追加時に、既存の動作を壊すコードを生成することがあります
   * **対策:** テストコードを書かせる、ADRで制約を明記する

4. **ドキュメントとコードの乖離**
   * 実装が進むと、仕様書が古くなることがあります
   * **対策:** 定期的に「ドキュメントを更新して」と指示

---

### 4-1. 2ページ目

5. **チームメンバ複数人で作業した場合**
   * チームに導入するための壁はまだまだあると感じます
   * **対策:** ルールを統一してドキュメント化する、メンバに経験を積ませる

#### 人間の役割は依然として重要

* **AIは「提案者」、人間は「意思決定者」**
* 仕様のレビュー、アーキテクチャの方向性、セキュリティ判断は人間が行います
* 「AIのためのマネージャー」になる体験です

**非エンジニア向けメッセージ:**
* 生成AIは「魔法」ではなく「優秀だけど指示待ちの新人」です
* 正しく導けば強力ですが、放置すると暴走します

* Free Tierがあり、個人開発なら無料で十分使えます
